#### DB설계 및 반정규화

- 적절한 데이터 타입 선택

  - 저장용량
  - 가능한 것 중 가장 작은 용량의 것을 선택
  - ex) 비밀번호는 몇으로 해야할까? 암호화? -> 단방향[V]= 비밀번호를 까서 원본이 뭔지 알아야할 필요가 없기 때문(복호화 필요 없음)
    -> 즉, sha256이면 길이가 해당 규격에 맞게 나오기 때문에 그 길이에 맞춰서 해야한다.
  - 양방향일 때는 클라이언트에서 암호화를 해야하는가? (X) 키가 노출된다.
  - ex) BigInt를 사용하는 것? -> 정말 21억을 넘어갈 수 있는가?

- 정규화 : 테이블 간 중복된 데이터를 최소화
  - 저장용량의 최적화 가능
  - 무결성 유지
  - JOIN을 많이 사용하게 되어서 성능이 떨어질 수 도 잇음
- 반정규화
  - 정규화에 위배되는 행위를 의도적으로 하는 것
  - 성능향상, 운영의 단순화
  - 정규화 이후 수행하라!
  - 대량의 범위를 자주 처리해야 하는 경우
  - 특정 범위의 데이터만 자주 처리하는 경우
  - 요약/ 집계 정보가 자주 요구되는 경우
  - JOIN으로 인한 성능 저하가 예상될 때
  - 테이블 병합(1:1, 1:N)
    - 1:1 테이블이 필요한 경우? -> 접근 권한을 줄 때 필요한 경우가 있음
  - 수퍼/서브 타입 테이블 병합 , 개인/기업 회원을 통합할 것이냐? 나눌 것이냐
  - 샤딩
